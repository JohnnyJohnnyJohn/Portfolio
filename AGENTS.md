# Repository Guidelines

## Project Structure & Module Organization

This is a Bun-powered TanStack Start React app. Application code lives in `src/`: route files are in `src/routes`, shared helpers are in `src/lib`, router setup is in `src/router.tsx`, and global Tailwind CSS is in `src/styles.css`. Static assets and web metadata live in `public/`. `src/routeTree.gen.ts` is generated by TanStack Router; do not edit it manually.

Use the configured aliases for imports where practical: `#/*` and `@/*` both point to `src/*`. Shadcn-style component aliases are reserved in `components.json` (`#/components`, `#/components/ui`, `#/hooks`) even if those folders are not present yet.

## Build, Test, and Development Commands

- `bun install`: install dependencies from `bun.lock`.
- `bun --bun run dev`: start the Vite dev server on port `3000`.
- `bun --bun run generate-routes`: regenerate `src/routeTree.gen.ts` after route changes.
- `bun --bun run build`: create a production build.
- `bun --bun run preview`: serve the built app locally for inspection.
- `bun --bun run test`: run the Vitest test suite once.
- `bun --bun run lint`, `format`, or `check`: run Biome linting, formatting, or both.

## Coding Style & Naming Conventions

TypeScript is strict (`noUnusedLocals`, `noUnusedParameters`, and related checks are enabled). Prefer TSX for React views and colocate route-specific logic with its route file. Route filenames under `src/routes` define URLs; keep `__root.tsx` for root layout concerns.

Biome is the formatter and linter. It uses tabs for indentation and double quotes for JavaScript/TypeScript formatting. Keep imports organized with Biome, and avoid manual edits to generated files.

## Testing Guidelines

Vitest is configured as the test runner, with React Testing Library and jsdom available. Add tests near the code they cover using names like `component.test.tsx` or `utils.test.ts`. Focus tests on route behavior, rendering states, and utility edge cases. Run `bun --bun run test` before opening a PR; run `bun --bun run check` for style and lint validation.

## Commit & Pull Request Guidelines

This repository currently has no Git commits, so no local commit convention is established. Use short, imperative commit subjects such as `Add portfolio route` or adopt Conventional Commits like `feat: add project gallery`.

PRs should include a concise description, linked issue when applicable, test results, and screenshots or screen recordings for visible UI changes. Note any route generation or configuration changes explicitly.
